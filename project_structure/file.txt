SECTION 2 — WHERE TO START
────────────────────────────────────────────────────────────────────────────────

Follow the phase plan from CLAUDE.md. Do NOT skip phases or jump ahead.
Each phase builds on the previous one. Attempting Phase 2 (audio) before
Phase 1 (database + auth) is stable will waste days debugging integration issues.

START HERE → Phase 1, Step 1: Monorepo scaffold

The correct development order within Phase 1 is:

  1. Monorepo scaffold           ← you are here
  2. Docker + database up
  3. Prisma schema + migration
  4. Auth API (login/refresh/logout)
  5. JWT middleware + role guards
  6. Admin course CRUD (text only, no audio)
  7. Admin student management
  8. Student login + subject list + chapter list
  9. BullMQ + Redis wired up (queue ready, no jobs yet)

Only after ALL of Phase 1 is working and tested → move to Phase 2 (audio).

The question trigger flow works like this:
  1. POST /student/progress/frame returns shouldTriggerQuestion: true
  2. Frontend polls GET /student/chapters/:id (or use WebSocket in Phase 4)
     to detect a new question has been saved for this student
  3. playerStore.showQuestion(question) is called
  4. QuestionOverlay renders
  5. Student taps an option
  6. useAnswerQuestion() POSTs the answer, gets isCorrect + iqDelta back
  7. Overlay shows result, "Continue" button calls dismissQuestion()
  8. Audio resumes from the next frame

NOTE: The simplest polling approach for Phase 3 is to check for a
pending question after each frame completes. You can upgrade to
WebSockets in Phase 4 if needed.

SECTION 4 — CODING RULES TO FOLLOW
────────────────────────────────────────────────────────────────────────────────

1. COPY FILES FROM project_structure/ — do not rewrite from scratch.
   The skeletons already have the correct types and logic. Your job is to
   wire them together and fill in any TODO comments.

2. ALWAYS start the database before the API.
   docker-compose up -d     (run this first every dev session)

3. ALWAYS run prisma generate after any schema change.
   npx prisma migrate dev --name <describe_change>
   npx prisma generate

4. TEST each API endpoint with Postman before building the UI for it.
   UI bugs are harder to debug. Confirm the API works first.

5. NEVER block the audio loop.
   LLM and TTS calls must be async/non-blocking.
   If generateQuestion() returns null → skip the question silently.
   If audio fails to load → advance to next frame automatically.

6. NEVER expose JWT_ACCESS_SECRET or AWS keys in the frontend.
   All secrets live in api/.env only.
   VITE_* variables are public — only put the API base URL there.

7. ENV VARS checklist before first run:
   □ DATABASE_URL          (from docker-compose)
   □ REDIS_URL             (from docker-compose)
   □ JWT_ACCESS_SECRET     (generate: openssl rand -hex 32)
   □ JWT_REFRESH_SECRET    (generate: openssl rand -hex 32)
   □ ANTHROPIC_API_KEY     (from console.anthropic.com)
   □ ELEVENLABS_API_KEY    (from elevenlabs.io — skip in Phase 1)
   □ AWS_*                 (from AWS console — skip in Phase 1)
